# Evaluate Mutations Against Test Suite

You are a mutation testing evaluator. Your job is to run previously generated mutations against the test suite, classify each as killed or survived, and produce an **HTML report**.

## Arguments

- `$ARGUMENTS` — parse the following named arguments:
  - `input` (required): path to the mutations JSON file (generated by `/generate-mutations`)
  - `output` (optional): path to the HTML report file
  - `parallelism` (optional): number of parallel test executions (default: 4)

## Step 0: Parse Arguments

1. Parse `$ARGUMENTS` into named parameters. Extract `input`, `output`, `parallelism`.
2. If `input` is missing, stop and ask the user for the mutations JSON file path.
3. If `output` is NOT provided, default to `~/Desktop/mutation-report.html`.
4. If `parallelism` is NOT provided, default to `4`.

## Step 1: Read Mutations File

1. Read the JSON file at `input` path.
2. Validate it has the expected structure:
   ```json
   {
     "commit": "<hash>",
     "generated_at": "<timestamp>",
     "packages": [...],
     "mutations": [...]
   }
   ```
3. Extract the `commit` hash and the `mutations` array.
4. If the file is invalid or empty, stop and report the error.

## Step 2: Infer Package/Module Per Mutation

For each mutation, infer the affected package/module from the mutation's `file` path. Do NOT require the user to specify packages.

### Inference Strategy

1. **Path-based inference** — match the file path against known project structure patterns:
   - `CommonSwift/<PackageName>/Sources/...` → Swift package, use `swift test` in `CommonSwift/<PackageName>/`
   - `Packages/<PackageName>/Sources/...` → Swift package, use `swift test` in `Packages/<PackageName>/`
   - `Core/<ModuleName>/...` → Core module
   - `ts-packages/<PackageName>/...` → TypeScript package, use its test script
   - `CrossplatformWeb/<path>/...` → Web package
   - `iOS/<path>/...` → iOS module, use `xcodebuild test` with the appropriate scheme
   - `Multiplatform/<path>/...`, `Web/<path>/...` → Kotlin, use gradle test tasks

2. **Package.swift / package.json detection** — walk up from the mutation file to find the nearest `Package.swift` or `package.json`. That directory is the package root.

3. **Build config fallback** — if no obvious package boundary, look for `BUILD`, `CMakeLists.txt`, `Cargo.toml`, or similar build files.

4. **Store the inferred package name** in each mutation result as `package`.

## Step 3: Execute Mutations

For EACH mutation in the mutations array:

### 3a. Apply Mutation in Isolation

1. Read the original file content at the mutation's `file` path.
2. Store the original content in memory (do NOT rely on git state).
3. Apply the mutation: replace the `original` code at the specified `location` (line, column) with the `mutated` code.
4. Write the modified file.

### 3b. Run Tests

1. Based on the inferred package (Step 2), run the appropriate test command:
   - **Swift packages** (`CommonSwift/`, `Packages/`): `swift test` in the package directory
   - **TypeScript packages**: `npm test` or `yarn test` or `pnpm test` in the package directory
   - **Kotlin modules**: `./gradlew :module:test`

2. Capture:
   - **Exit code**: 0 = all tests pass, non-zero = at least one test failed
   - **Execution time** in milliseconds
   - **Failed test names**: parse test output to extract which tests failed (if any)
   - **stdout/stderr**: for debugging if needed

### 3c. Classify Mutation

- **Killed**: test exit code != 0 (at least one test failed due to the mutation)
- **Survived**: test exit code == 0 (all tests passed despite the mutation — indicates weak test coverage)

### 3d. Restore Original Code

1. Write the original file content back **immediately** after capturing test results.
2. Verify the file is restored by comparing content.
3. This step is CRITICAL — never leave mutated code in the working directory.

### 3e. Record Result

Store the evaluation result for this mutation:
```json
{
  "id": "<mutation-id>",
  "file": "<file-path>",
  "location": { "line": N, "column": N },
  "type": "<mutation-type>",
  "category": "<standard|advanced>",
  "original": "<original-code>",
  "mutated": "<mutated-code>",
  "status": "killed|survived",
  "tests_failed": ["testName1", "testName2"],
  "execution_time_ms": 120,
  "package": "<inferred-package>"
}
```

## Step 4: Performance Optimizations

- Run up to `parallelism` mutation evaluations concurrently using the Agent tool.
- **IMPORTANT**: Parallel mutations MUST target different packages or modules. If two mutations target the same file, they MUST run sequentially to avoid conflicts.
- Group mutations by file, then by package.
- Within a file, run mutations sequentially; across different files/packages, run in parallel.
- Reuse build artifacts between mutations in the same package.

3. For the full results table, include ALL mutations. Each `<tr>` must have `data-*` attributes for filtering/sorting:
   ```html
   <tr data-id="..." data-package="..." data-file="..." data-type="..." data-status="..." data-time="...">
   ```

4. Use `<code>` tags for original/mutated code snippets.
5. Use badge classes for status: `badge-killed` or `badge-survived`.

## Safety Requirements

- NEVER leave modified files in the working directory. Always restore originals.
- NEVER commit or push any changes.
- Store original file content in memory BEFORE applying any mutation.
- Restore the file IMMEDIATELY after test execution, even if tests crash.
- If a mutation targets a file that doesn't exist, skip it and log a warning.
- If the test command fails to execute (not test failure, but execution error), log it and continue.

## Execution Strategy

Use the Agent tool to parallelize work:
1. Group mutations by file to avoid parallel writes to the same file.
2. Spawn agents per package (or per file group) for parallel evaluation.
3. Each agent handles: apply mutation → run tests → capture results → restore file.
4. Aggregate all results into the final HTML report.
5. CRITICAL: Two mutations on the same file must NEVER run concurrently.

## Caching (Optional)

- Cache directory: `.claude/mutation-eval-cache/`
- Cache key: `<commit-hash>-<mutation-id>`
- Before running tests for a mutation, check if the result is already cached for this commit.
- If cached, reuse the result without re-running tests.
- Invalidate cache if any source file in the package has changed since the cache entry was written.

## Important Notes

- This is a **test execution and reporting** task, not just analysis.
- You MUST actually apply mutations, run tests, and capture real results.
- You MUST generate the HTML report file.
- You MUST print the console summary.
- Survived mutations are the PRIMARY output — they indicate weak test coverage.
- The HTML report must be visually clean and immediately useful without any external tools.
- Prioritize correctness and isolation over speed.
